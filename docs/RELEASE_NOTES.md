# Release Notes

> **Client-facing record of all user-visible changes.**
> Written for end users — not developers. Keep language clear, non-technical, and benefit-oriented.
> **Newest entries at TOP.** Each change gets its own timestamped entry, even if multiple changes land on the same day.

---

### ⚠️ Content Rules

These rules are **non-negotiable** and must be followed for every entry:

1. **CLIENT-FACING ONLY** — Only include changes that affect the end-user experience. If a user would never notice it, it does not belong here.
2. **NO admin, internal, or backend-only changes** — Infrastructure, refactors, developer tooling, CI/CD, internal dashboards, and anything only visible to developers or admins must be excluded.
3. **NO sensitive content** — Never expose API keys, internal URLs, infrastructure details, database schemas, vendor names, or implementation specifics.
4. **Security items → "Security Enhancements" ONLY** — If a security-related change was made, include a single line such as "We've made security enhancements to better protect your data." Do NOT describe the vulnerability, the vector, the fix, or any technical detail whatsoever.
5. **Every change = a new entry** — Even if five changes happen in the same session on the same day, each one gets its own timestamped entry. No combining, no batching.
6. **Plain language** — Write as if explaining to a non-technical customer. Avoid jargon. Focus on the benefit or improvement they'll experience.
