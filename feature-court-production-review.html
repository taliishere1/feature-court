Vercel Build & Deploy Verdict

✓

The project BUILDS on Vercel as it stands.

A clean production build was run locally and succeeded: next build on Next.js 16.2.9 using Turbopack (the default and Vercel-recommended bundler in Next 16), TypeScript type-checking passed, and all 13 routes were generated. There is no custom webpack config, so the Turbopack-by-default behavior in Next 16 will not error on Vercel. No vercel.json is required for this app.

!

Required before the deployed site will work: set environment variables in Vercel.

The build succeeds without them, but at runtime the browser Supabase client (src/lib/supabase.ts) returns null if the public env vars are missing, and pages call it with a non-null assertion (supabase!) — so the live site would crash even though it built. In the Vercel Project → Settings → Environment Variables, add:

NEXT_PUBLIC_SUPABASE_URL NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY

Only the publishable key belongs in NEXT_PUBLIC_ vars (it is meant to be public). The OpenAI key and any secret keys must remain Supabase Edge Function secrets — never add them to NEXT_PUBLIC_ vars.

Medium

IDSevAreaFinding & Recommendation
M1MediumVercel / Perf OG image asset is 6.97 MB and read+base64-encoded on every request. public/images/seal.png is ~7 MB; src/app/og/route.tsx reads it via fs, base64-encodes it (~+33% in memory), then renders it at 180×180. This wastes memory/CPU per OG hit and risks an ENOENT on Vercel if the file isn't traced into the function bundle. Fix: downsize the seal to a ~180–360px PNG (<50 KB), and add outputFileTracingIncludes in next.config.ts to guarantee inclusion (or fetch the asset by URL).
M2MediumBackend / Cost Stage pages regenerate content on every visit (no idempotency). prosecution, defense, and cross pages invoke their edge function unconditionally on mount; a refresh or back-navigation re-runs gpt-5.4 generation, costing tokens, risking content drift, and advancing the previous_response_id chain pointer. The ruling page already does this correctly (it checks for existing verdicts first). Fix: mirror the ruling page — read the row, skip the invoke if that stage's content already exists.
M3FlagSecurity / Auth Edge functions are publicly invocable (no in-function auth). With verify_jwt=false and no custom guard, anyone with the (public) URL + publishable key can invoke the AI functions, each costing a gpt-5.4 call — a cost-abuse / DoS surface. Per your direction this is recorded as an insight, not a heavy deduction. Recommendation: handle auth inside the functions — verify a short-lived JWT with jose (jwtVerify against your JWKS / signing key), and/or add rate limiting or a Turnstile/hCaptcha check. This keeps you fully off legacy JWT verification while closing the abuse gap.

Overload / CPU assessment

  • No connection-exhaustion risk: the app and functions use the Supabase Data API (PostgREST over HTTP), not raw Postgres connections — pooling is handled by the platform.
  • No per-row RLS cost: policies use constant true (no auth.<fn>() calls evaluated per row), so RLS adds no measurable CPU.
  • Query plans are index scans for every app query; the table is tiny and writes are single-row upserts/updates.
  • The real throughput/cost driver is OpenAI latency & the open-function abuse surface (M3), not the database. Per-stage function times measured 0.6s–17.5s; the client polling timeout (60s) sits comfortably above the worst case.
